# Copilot Customizations

GitHub Copilot agents, skills, and prompts for this project.

## Setup

The customization files are located in:

```
.github/
├── agents/
│   ├── coder.agent.md
│   ├── copilot-evaluator.agent.md
│   ├── designer.agent.md
│   ├── orchestrator.agent.md
│   ├── planner.agent.md
│   └── strapi-copilot-agent.agent.md
├── prompts/
│   ├── code-review.prompt.md
│   ├── commit.prompt.md
│   └── strapi-init.prompt.md
├── skills/
│   ├── copilot-best-practices-agents/SKILL.md
│   ├── copilot-best-practices-instructions/SKILL.md
│   ├── copilot-best-practices-prompts/SKILL.md
│   ├── copilot-best-practices-skills/SKILL.md
│   ├── copilot-customization-guide/SKILL.md
│   ├── strapi-v4/SKILL.md
│   └── strapi-v5/SKILL.md
├── strapi-config.example.json
├── strapi-config.json          ← git-ignored, your local config
└── strapi-schemas.json         ← git-ignored, generated by /strapi-init
```

No additional installation is required — GitHub Copilot automatically detects agents and skills from the `.github/` directory.

To reduce IDE schema warnings as GitHub Copilot evolves, these customization files intentionally stick to the smallest stable frontmatter surface where possible:

- Prompts use `description`, `agent: 'agent'`, and `tools`
- Agents use `description` and `tools`, with `name` only when it helps discoverability
- Optional fields like `model` are avoided unless they are clearly needed

## Strapi Workflow

Everything you need to query and work with Strapi APIs. Follow these steps to get started, then use the agent for day-to-day queries.

### 1. Configure your connection

Copy the example config and fill in your values:

```bash
cp .github/strapi-config.example.json .github/strapi-config.json
```

Then edit `.github/strapi-config.json`. You can configure one or multiple Strapi instances:

**Single instance:**

```json
{
  "instances": {
    "main": {
      "baseUrl": "http://localhost:1337",
      "token": "your-api-token"
    }
  },
  "default": "main"
}
```

**Multiple instances:**

```json
{
  "instances": {
    "blog": {
      "baseUrl": "http://localhost:1337",
      "token": "blog-token"
    },
    "cms": {
      "baseUrl": "http://localhost:1338",
      "token": "cms-token"
    }
  },
  "default": "blog"
}
```

| Field | Description |
|-------|-------------|
| `instances` | Named Strapi instances, each with its own `baseUrl` and `token` |
| `instances.<name>.baseUrl` | The Strapi API URL for this instance |
| `instances.<name>.token` | Optional API token for this instance |
| `default` | Which instance to use when the user doesn't specify one |

> **Note:** `strapi-config.json` is git-ignored to prevent leaking tokens. Only the `.example.json` template is committed.

### 2. Initialize your schemas

Run the init prompt so the agent learns your content types:

```
/strapi-init
```

This fetches content-type schemas from **all configured instances** and saves them to `.github/strapi-schemas.json`. After this, the agent knows your tables, fields, and relations across all instances.

> **Requires an admin API token.** A regular API token won't have access to the content-type-builder endpoint. Generate one in Strapi under Settings > API Tokens.

Re-run `/strapi-init` whenever you add or modify content types.

### 3. Query your data

Use the Strapi agent to fetch data. It uses the default instance unless you specify one:

```
@strapi-copilot-agent Fetch all articles
```

```
@strapi-copilot-agent Get all products where price is greater than 50
```

```
@strapi-copilot-agent Get articles with category and author populated
```

**Query a specific instance:**

```
@strapi-copilot-agent Fetch all pages from cms
```

```
@strapi-copilot-agent Get articles from blog and pages from cms
```

```
@strapi-copilot-agent What content types are available?
```

### 4. Version-specific help

The agent auto-detects your Strapi version, but you can also ask directly:

```
@strapi-copilot-agent I'm on Strapi v4 — show me how to query articles with nested category
```

```
@strapi-copilot-agent I'm migrating from v4 to v5 — what changes do I need for my article queries?
```

```
@strapi-copilot-agent How do I access a relation field in Strapi v5?
```

### Authentication

Set the `token` field in `.github/strapi-config.json` and the agent includes it automatically. You can also pass it inline:

```
@strapi-copilot-agent Fetch articles using bearer token abc123xyz
```

### What the Strapi agent can and cannot do

| Can do | Cannot do |
|--------|-----------|
| Fetch data (GET requests) | Create, update, or delete data |
| Filter, sort, and paginate | Authenticate on your behalf |
| Populate relations | Access admin-only endpoints |
| Detect Strapi version (v4/v5) | Modify content types |
| Suggest fields based on schemas | |

### Strapi v4 vs v5 at a glance

| Area | Strapi v4 | Strapi v5 |
|------|-----------|-----------|
| Response format | Nested `data.attributes` | Flat on `data` |
| Entity ID | Numeric `id` | `documentId` (string) |
| Fetch by ID | `GET /api/articles/1` | `GET /api/articles/abc123` |
| Access a field | `data.attributes.title` | `data.title` |

## Copilot Evaluator

An agent that reviews customization files and provides clear, junior-friendly feedback. It checks classification, structure, and best practices — then explains issues in plain language with actionable fixes.

Before evaluating, it now fetches the latest official GitHub Copilot documentation from `docs.github.com` and uses that as the source of truth if local guidance is outdated.

When it finds an issue, it also provides real follow-up resources:

- Official GitHub Docs links for the relevant file type or schema rule
- Matching local guidance files from `.github/skills/`

### Review a file

```
@copilot-evaluator Review .github/agents/strapi-copilot-agent.agent.md
```

```
@copilot-evaluator Is this skill set up correctly? .github/skills/strapi-v5/SKILL.md
```

### Validate something you just created

```
@copilot-evaluator I created a new skill at .github/skills/my-api/SKILL.md — does it look right?
```

```
@copilot-evaluator I added a prompt at .github/prompts/generate-tests.prompt.md — is it the correct type?
```

### Check if something is the wrong type

```
@copilot-evaluator Should this be a skill or an agent? .github/skills/database-helper/SKILL.md
```

```
@copilot-evaluator I have coding standards in a skill — should it be instructions instead?
```

### Review multiple files at once

```
@copilot-evaluator Review all files in .github/agents/
```

```
@copilot-evaluator Check all our skills for best practices
```

### What the evaluator checks

1. **Correct type** — is this file an instruction, skill, prompt, or agent? Should it be something else?
2. **Required fields** — does the YAML header match the latest GitHub Copilot schema for that file type?
3. **Effective description** — will Copilot know when to load this?
4. **Content quality** — is it well-structured with examples? Are constraints clear?
5. **Current docs alignment** — does it still match the latest official documentation?
6. **Useful resources** — does the report point you to the right official docs and local guidance files?
7. **Common mistakes** — does it match known anti-patterns?

### Example output

The evaluator produces reports like this:

```
## Evaluation: database-helper/SKILL.md

### Summary
This file is classified as a skill but it needs direct tool access — it should be an agent.

**Current type:** Skill
**Should be:** Agent
**Overall:** Wrong type

---

### What's working
- Good description that explains when to use it
- Well-structured content with examples

### What needs fixing

#### Should be an agent, not a skill
**What's wrong:** This file lists direct tool access and defines
behavioral constraints ("only run SELECT queries"). That fits GitHub Copilot's
agent model better than a skill.
**Why it matters:** As a skill, the tools won't work and Copilot will
ignore the constraints.
**How to fix:**
1. Move the file from `.github/skills/database-helper/SKILL.md`
   to `.github/agents/database-helper.agent.md`
2. Change the YAML header to use agent format
```

## Best Practices Skills

Reference skills the evaluator uses — also available to Copilot in any conversation when the topic comes up.

| Skill | What it covers |
|-------|---------------|
| `copilot-best-practices-agents` | Agent file structure, required fields, tools config, when to use vs other types |
| `copilot-best-practices-skills` | SKILL.md format, naming rules, description writing, security warnings |
| `copilot-best-practices-prompts` | Prompt file structure, `agent: 'agent'` and tools config, variables, task vs reference |
| `copilot-best-practices-instructions` | Repo-wide vs path-specific, glob patterns, concise rules, conflict avoidance |
| `copilot-customization-guide` | Decision framework for choosing between all four types |

These skills load automatically when relevant — you don't need the evaluator agent to access them. For example, if you ask Copilot "how do I create a new skill?", the best practices skill will load on its own.

## Quick Reference

### When to use what

| I need to... | Use | Think of it as... |
|--------------|-----|-------------------|
| Set a rule for every interaction | Instructions | House rules |
| Add specialized instructions/resources Copilot loads when relevant | Skill | A domain playbook |
| Create a repeatable task I trigger manually | Prompt | A recipe card |
| Build a specialist with tools and constraints | Agent | A hired expert |

### File locations

**Instructions:**

| Location | Scope |
|----------|-------|
| `.github/copilot-instructions.md` | Repo-wide |
| `.github/instructions/**/*.instructions.md` | Path-specific (uses `applyTo` glob) |
| `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` (repo root) | Agent instructions |
| `$HOME/.copilot/copilot-instructions.md` | Personal (all repos) |
| `COPILOT_CUSTOM_INSTRUCTIONS_DIRS` env var | Additional directories |

**Skills** (file must be named `SKILL.md` in its own subdirectory):

| Location | Scope |
|----------|-------|
| `.github/skills/<name>/SKILL.md` | Project |
| `.claude/skills/<name>/SKILL.md` | Project (alternative) |
| `.agents/skills/<name>/SKILL.md` | Project (alternative) |
| `~/.copilot/skills/<name>/SKILL.md` | Personal |
| `~/.claude/skills/<name>/SKILL.md` | Personal (alternative) |
| `~/.agents/skills/<name>/SKILL.md` | Personal (alternative) |

**Prompts:**

| Location | Scope |
|----------|-------|
| `.github/prompts/*.prompt.md` | Project |

Prompt files are typically invoked with `/prompt-name`.

**Agents:**

| Location | Scope |
|----------|-------|
| `.github/agents/*.agent.md` | Project |
| `~/.copilot/agents/*.agent.md` | Personal (overrides project if same name) |
